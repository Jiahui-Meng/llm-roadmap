# Day 71 — Agent Tools Implementation

## 1. 为什么 Agent 工具实现不是“把 API 接上去”这么简单

到了 Day 71，很多人会觉得这一天就是写几个函数：
- search
- scrape
- retrieve
- summarize

但如果只是把工具接出来，而没有考虑它们在 workflow 中的角色，那这些工具很容易变成：
- 接口风格不一致
- 错误处理混乱
- 输入输出边界不清楚
- 很难复用

所以 Day 71 的核心不是“有工具”，而是：

> **工具是否能成为一个稳定的系统能力单元。**

---

## 2. 工具在 agent 里到底扮演什么角色

LLM 本身擅长：
- 语言理解
- 推理
- 总结
- 解释

但不擅长：
- 实时搜索世界信息
- 读取外部网页
- 精准访问数据库
- 稳定执行多步操作

工具的作用就是把这些能力补上。

所以工具不是附加件，而是：

> **LLM 接触外部世界的接口。**

---

## 3. 一个好的工具设计应该满足什么

至少应满足：

### 1）输入清楚
工具到底吃什么？

### 2）输出清楚
工具到底返回什么结构？

### 3）错误可处理
失败时返回什么？抛错还是 fallback？

### 4）结果可被下游继续消费
输出结构最好稳定、可解析、可追踪。

这意味着 Day 71 的重点不是“功能强”，而是：

> **工具接口契约要稳定。**

---

## 4. 为什么 search / scrape / retrieve / summarize 是一个经典组合

这四个工具基本构成了 research agent 的最小工具链：

### search
发现候选外部信息源

### scrape
从候选网页中提取正文

### retrieve
从本地知识库中召回相关内容

### summarize
把多来源证据合成结构化摘要

这一组合的意义在于：
- 同时覆盖外部世界和内部知识
- 让 agent 既能“找”，也能“读”，还能“整合”

所以 Day 71 其实是在搭一个研究型 agent 的感知和行动底座。

---

## 5. 为什么工具最好先用 mock 版本跑通

工程上一个常见好习惯是：

> **先让工具接口稳定，再替换真实后端。**

因为如果你一开始就接真实 API，会同时面对：
- 网络问题
- 权限问题
- 速率限制
- 格式不稳定
- workflow 逻辑问题

这样很难定位故障。

所以先用 mock 版本有两个好处：
- 把 workflow 先跑顺
- 把工具接口先定稳

这和前面很多地方的原则完全一致：

> **先跑通，再增强。**

---

## 6. 工具实现为什么也需要 observability

工具最怕“静默失败”。

比如：
- search 返回空结果
- scrape 抓到了导航栏而不是正文
- retrieve 命中了错误知识库
- summarize 漏掉关键引用

如果你不记录中间输入输出，就很难知道问题出在哪。

所以一个成熟的工具实现最好至少记录：
- 输入参数
- 输出摘要
- 错误信息
- 延迟

这也为 Day 73 的 trace logging 铺路。

---

## 7. 今天最该记住的 5 句话

1. **Agent 工具实现的重点不是“接上 API”，而是接口稳定。**
2. **工具是 LLM 接触外部世界的接口。**
3. **输入、输出、错误处理和可消费性是工具设计核心。**
4. **search / scrape / retrieve / summarize 是研究型 agent 的经典最小工具链。**
5. **先用 mock 跑通接口，再接真实后端，是更合理的工程路径。**

> Agent Tools Implementation 的本质，是把搜索、抓取、检索和摘要这些能力封装成输入输出清晰、错误可处理、结果可复用的稳定系统接口，从而让 LLM 真正具备与外部世界交互的能力。
