# Day 57 — Fine-tuning Task Selection

## 1. 为什么微调项目第一步不是“立刻开训”

很多人一进入 LoRA / QLoRA 阶段，会自然地想：
- 先找个模型
- 先找点数据
- 直接跑训练

但现实里，微调项目最容易失败的地方，往往不是训练脚本，而是：

> **你一开始选的任务就不适合做。**

如果任务选错了，后面会出现一连串问题：
- 数据太难收
- 标注标准不清楚
- 评估做不起来
- prompt baseline 已经够好，微调收益不明显
- 训练后很难证明有提升

所以 Day 57 的重点不是“选一个听起来高级的任务”，而是：

> **选一个既能做、又能评估、还能展示价值的任务。**

---

## 2. 一个好的 fine-tuning 任务应该满足什么

好的任务一般至少满足 4 个条件：

### 1）范围够窄
任务不能太泛，否则：
- 数据需求爆炸
- 标注标准模糊
- 微调目标不清晰

### 2）数据能拿到
如果数据收集成本过高，项目很容易停在半路。

### 3）输出容易评估
如果结果没有明确判断标准，微调效果很难验证。

### 4）微调有希望带来增益
如果 prompt baseline 已经几乎完美，那微调的价值就很小。

所以任务选择的本质是：

> **在“可执行性”和“展示价值”之间找平衡。**

---

## 3. 为什么“任务越大越好”是错觉

初学者很容易被大而全的任务吸引，比如：
- 通用聊天微调
- 全领域问答
- 大规模代码助手

这些任务听起来很酷，但问题是：
- 数据量大
- 质量难控
- eval 困难
- 很难在有限资源下完成

所以 Day 57 更推荐的是：
- support FAQ assistant
- style adaptation
- structured extraction
- domain QA
- evidence summarization

因为这些任务：
- 范围清晰
- 训练目标明确
- before/after 对比更容易做

这也是学习型项目最合理的切入方式。

---

## 4. 为什么 baseline prompt 很重要

在正式微调前，你应该先问一个问题：

> **如果只靠 prompt，这个任务现在能做到几分？**

这是非常关键的，因为：
- 如果 baseline prompt 已经很好，微调收益可能有限
- 如果 baseline prompt 很差，但任务又有稳定模式，微调往往更有价值

所以 task selection 不能脱离 baseline。

你选的不是“一个理论上可微调的任务”，而是：

> **一个值得微调的任务。**

---

## 5. 为什么 Day 57 是整个 LoRA 项目的成败点之一

LoRA / QLoRA 后面的所有步骤都依赖任务选择：
- Day 58：怎么收数据
- Day 59：怎么转 instruction format
- Day 60：怎么设训练目标
- Day 61：怎么做 eval

如果任务选得模糊，后面每一步都会变得模糊。

所以 Day 57 的作用很像：

> **为整个微调项目定义清晰的问题边界。**

---

## 6. 什么样的任务最适合作为作品集项目

如果你希望这个项目未来能进：
- GitHub
- 简历
- 面试讲解

那任务最好满足：
- 场景真实
- 数据来源合理
- 输出有可评估标准
- 微调前后差异可展示

比如：
- JD 技能抽取
- 证据型摘要
- 结构化报告生成
- 某领域 QA 风格适配

这些任务通常比“训练一个万能助手”更适合展示工程能力。

---

## 7. 今天最该记住的 5 句话

1. **微调项目最容易失败的地方，往往是任务选错。**
2. **好的任务必须同时满足可执行、可评估和有增益空间。**
3. **任务范围越窄，越容易跑通完整实验闭环。**
4. **baseline prompt 表现决定这个任务是否值得微调。**
5. **Day 57 是整个 LoRA 项目的问题定义日。**

## 9. 一句话总结

> Fine-tuning task selection 的本质，是为整个 LoRA/QLoRA 项目选定一个边界清晰、数据可得、结果可评估、并且真正值得微调的问题，从而保证后续训练和评估能够形成完整闭环。
