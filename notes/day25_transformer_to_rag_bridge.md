# Day 25 — Transformer to RAG Bridge

## 1. 为什么会从 Transformer 走向 RAG

到 Day 24 为止，你已经学了很多模型内部机制：
- attention
- self-attention / multi-head attention
- Transformer block
- masks
- causal LM
- context window
- KV cache
- modern LLM block

这些知识让你理解了：

> **模型为什么会生成、为什么会推理、为什么在很多任务上很强。**

但一旦模型进入真实应用，就会暴露几个根本边界：
- 参数知识更新不及时
- 容易幻觉
- 上下文再长也有限
- 很难对企业私有知识做稳定对齐

这就是为什么你会自然从 Transformer 走向 RAG。

RAG 不是否定 Transformer，而是承认：

> **模型本体很强，但它仍然需要外部知识系统来补足现实世界的动态性与可验证性。**

---

## 2. Transformer 的边界到底在哪里

一个大模型再强，本质上仍然主要依赖两类知识来源：

### 1）参数化知识
也就是它在预训练 / 微调里学到的内容。

### 2）prompt 中提供的上下文
也就是你在推理时塞给它的内容。

这两种方式各有问题：

#### 参数化知识的问题
- 更新成本高
- 无法快速同步新信息
- 容易和真实世界时间脱节

#### Prompt 上下文的问题
- token 预算有限
- 不能总把所有资料塞进去
- 文档一多就会很贵、很乱

所以 Day 25 最重要的认识是：

> **Transformer 很强，但它不天然等于“随时可访问全世界知识”。**

---

## 3. RAG 在补什么

RAG（Retrieval-Augmented Generation）做的事情可以简单理解成：

```text
用户问题
-> 检索相关外部知识
-> 把检索结果放进上下文
-> 再由 LLM 生成回答
```

也就是说，RAG 不是只让模型“凭记忆回答”，而是先让系统：
- 找资料
- 筛资料
- 再回答

所以 RAG 补上的不是模型参数，而是：

> **模型与外部知识之间的连接机制。**

---

## 4. 为什么长上下文不能直接替代 RAG

很多人会问：

> 如果模型上下文已经很长了，还需要 RAG 吗？

通常仍然需要。

原因包括：
- 长上下文很贵
- 检索更适合从大文档库中先筛相关内容
- 模型并不总能均匀利用超长上下文
- 用户很多问题其实只需要极少量关键证据

所以 RAG 和长上下文的关系更像：
- 长上下文提供容纳能力
- RAG 提供相关性选择能力

也就是说：

> **长上下文解决“能放多少”，RAG 解决“该放什么”。**

---

## 5. 为什么 RAG 是系统能力，而不是模型能力

这也是 Day 25 的核心桥接点。

Transformer 学的是：
- token 如何交互
- 上下文如何建模
- 输出如何生成

RAG 引入的是：
- 文档摄入
- chunking
- embedding
- retrieval
- reranking
- grounding
- citation

这些东西不属于模型 block 内部，而属于：

> **模型外部的系统层。**

所以从 Transformer 到 RAG，你真正跨过的是一层抽象：

- 从模型结构理解
- 走向知识系统设计

---

## 6. 为什么 Day 25 是一个真正的“桥梁日”

前面你学的是：
- 模型为什么强
- 模型内部机制是什么

从 Day 25 开始，你会进入：
- chunking
- embeddings and retrieval
- reranking
- RAG v1 / v2
- evaluation

所以 Day 25 的作用不是教你一个新公式，而是帮你完成一次认知切换：

> **从“理解大模型”走向“构建基于大模型的知识系统”。**

这一步非常关键，因为很多真实工作岗位更看重的正是这层能力。

---

## 7. RAG 为什么会成为 LLM 工程的核心主题之一

因为现实里大量高价值场景都需要：
- 动态知识
- 私有知识接入
- 可验证回答
- 降低幻觉
- 可维护知识更新

例如：
- 企业知识库问答
- 文档助手
- 研究助手
- 医疗 / 法律系统
- 内部 SOP 问答

这些都不适合只靠模型参数记忆。

因此从 Day 25 开始，roadmap 进入的是：

> **LLM 工程里最常见、最有业务价值的一条主线。**

---

## 8. 今天最该记住的 5 句话

1. **Transformer 很强，但它的知识主要来自参数和 prompt。**
2. **参数知识不适合快速更新，也不天然可验证。**
3. **RAG 用外部检索系统把知识接到生成流程里。**
4. **长上下文不能替代 RAG，因为它不解决相关性筛选。**
5. **从 Transformer 到 RAG，是从模型能力走向知识系统能力。**

> 从 Transformer 走向 RAG，本质上是承认模型参数化知识和上下文容量都有现实边界，然后通过检索系统把外部、动态、可验证的知识接入生成流程，从而把“会回答”升级成“更会基于真实资料回答”。
