# Day 64 — Schema-Constrained Generation

## 1. 为什么只要求 JSON 还不够

Day 63 你已经知道，让模型输出 JSON 很有用。

但只说“请输出 JSON”仍然不够，因为：
- JSON 里可以缺字段
- 类型可以错
- 数组长度可能不符合要求
- 字段值可能超出约束

也就是说：

> **JSON 只是容器，schema 才是真正的结构约束。**

这就是 Day 64 要解决的问题。

---

## 2. schema-constrained generation 在做什么

它的目标是：
- 不只让模型输出“看起来像 JSON”
- 而是让输出满足预定义结构和约束

比如：
- 必须包含哪些字段
- 每个字段的数据类型是什么
- 数组长度限制是多少
- 字符串最大长度是多少
- 是否允许嵌套对象

所以 Day 64 的本质是：

> **把“结构化输出”从松散格式要求升级成可验证协议。**

---

## 3. 为什么 schema 对下游系统很重要

如果下游系统依赖模型输出，例如：
- 填表
- 结构化抽取
- tool arguments
- 工作流状态更新

那输出格式一旦漂移，就可能：
- 解析失败
- 字段缺失
- 类型错误
- 直接把整个 workflow 搞崩

schema 的价值就是减少这种不确定性。

换句话说：

> **schema 让 LLM 输出更像接口契约。**

---

## 4. schema 和 JSON 的区别

这是 Day 64 最重要的基础点之一。

### JSON
解决的是：
- 输出是不是结构化文本

### Schema
解决的是：
- 输出结构是否满足预定义要求

所以：
- JSON 是格式
- schema 是约束

如果没有 schema，JSON 依旧可能是工程灾难。

---

## 5. schema-constrained generation 最常见的场景

它非常适合：
- 信息抽取
- 表单生成
- 工具调用参数生成
- 分类标签输出
- 结构化摘要

这些任务共同特点是：
- 输出结构比文采更重要
- 下游代码要直接消费结果

所以 Day 64 是很多 agent / automation 场景的前置能力。

---

## 6. 为什么 Day 64 是从“可解析”到“可验证”的升级

Day 63 已经让输出可以 parse。

Day 64 则进一步要求：
- parse 之后还得合法
- 合法之后还得满足业务约束

这是一次很大的升级，因为系统关心的不只是：
- “我能不能把它读出来”

而是：
- “我能不能放心把它交给下一步流程”

所以 Day 64 的核心转变是：

> **从可解析输出，走向可验证输出。**

---

## 7. 今天最该记住的 5 句话

1. **JSON 只是格式，schema 才是真正约束。**
2. **schema-constrained generation 让输出更像接口契约。**
3. **对下游自动化系统来说，可验证比“看起来结构化”更重要。**
4. **schema 特别适合抽取、工具调用和 workflow 状态更新。**
5. **Day 64 标志着 structured generation 从可解析走向可验证。**
