# Day 61 — LoRA Eval Report

## 1. 为什么微调之后必须写 eval report

很多人跑完训练后，会很自然地停在：
- loss 降了
- 模型能输出了
- 看起来好像有点提升

但这还远远不够。

因为一个微调实验真正有价值，不在于“训完了”，而在于：

> **你能不能清楚说明它到底改善了什么，又带来了什么代价。**

这正是 Day 61 的任务。

---

## 2. eval report 在回答什么问题

一个好的 LoRA eval report 至少应该回答：
- 模型相比 baseline 提升了吗？
- 提升体现在哪些样例上？
- 哪些能力没有提升？
- 是否出现了新的退化？
- 这次训练值不值得继续做下一轮？

所以 eval report 不是形式文档，而是：

> **把训练结果变成工程判断的依据。**

---

## 3. 为什么 before / after 对比是核心

评估微调最直观的方式就是：
- 同一组 prompt
- baseline 模型输出
- finetuned 模型输出
- 人工或规则对比

如果没有 before / after，你就很难判断：
- 这次变化是不是训练造成的
- 提升是不是稳定的
- 风格变化是不是目标内的

所以 Day 61 的核心不是抽象指标，而是：

> **同题对比。**

---

## 4. eval report 不应该只写好消息

一个常见错误是：
- 只挑变好的案例
- 不写失败样例
- 不提副作用

这会让报告失去工程价值。

因为真实训练经常是：
- 某些方面变好了
- 某些方面没变
- 某些方面甚至变差了

所以一个有价值的 eval report 必须诚实展示：
- gains
- regressions
- trade-offs

这也是成熟工程思维的一部分。

---

## 5. LoRA eval 最应该看哪些维度

根据任务不同，至少可以看：
- instruction following fidelity
- 输出格式稳定性
- 领域正确性
- hallucination rate
- 风格一致性
- error type 分布

重要的是：

> **评估维度必须和 Day 57 选定的任务目标一致。**

否则报告会变得很空。

---

## 6. 为什么 Day 61 是 LoRA mini project 的收口日

Day 57–60 你做的是：
- 定义任务
- 准备数据
- 转格式
- 训练 run1

Day 61 要做的是：

> **把这条实验链路闭环。**

也就是说，从这一天开始，你不只是“做过一次训练”，而是拥有了：
- 一个任务
- 一套数据
- 一轮实验
- 一份可解释评估

这才叫完整项目。

---

## 7. eval report 和下一轮实验的关系

好的 eval report 不只是总结过去，还要指导下一步。

它应该帮助你判断：
- 是否继续这个任务
- 是否要补更多数据
- 是否要改 instruction format
- 是否要调整 LoRA 超参
- 是否需要更换 base model

所以 eval report 本质上也是下一轮实验的导航图。

---

## 8. 今天最该记住的 5 句话

1. **微调训练完成不等于项目完成，评估报告才让实验闭环。**
2. **before / after 同题对比是最核心的评估方式。**
3. **eval report 必须同时写 gains 和 regressions。**
4. **评估维度必须与原任务目标一致。**
5. **Day 61 的价值是把一次训练变成一轮完整实验。**

---

## 9. 今日任务

### 必做
1. 选一组固定 prompts 做 before / after 对比
2. 写出 3 个提升案例
3. 写出 3 个失败或退化案例
4. 写出下一轮训练最值得改进的 2 个方向

### 你要能回答的问题
1. 为什么 LoRA 训练后必须写 eval report？
2. before / after 对比为什么这么重要？
3. 为什么不能只写“好消息”？
4. eval 维度应该如何确定？
5. Day 61 为什么标志着 mini project 闭环？

---

## 10. 一句话总结

> LoRA eval report 的本质，是把一次训练的结果从“看起来有效”转变为“可对比、可解释、可复盘、可指导下一轮实验”的工程结论，从而真正闭合微调项目的实验回路。
