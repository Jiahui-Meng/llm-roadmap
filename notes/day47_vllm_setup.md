# Day 47 — vLLM Setup

## 1. 为什么 Day 47 不是简单“装个环境”

很多人看到 setup 这一天，会下意识觉得：
- 就是装个依赖
- 跑个命令
- 看到服务起来就完了

但在 LLM 工程里，setup 从来不是纯机械步骤。

因为你真正验证的不是“某条命令能不能执行”，而是：

> **你的本地或远程环境，是否真的具备跑大模型 serving 的最小条件。**

这包括：
- 驱动和 CUDA 是否匹配
- 显存是否够
- 模型格式是否兼容
- 推理框架和模型是否能正确握手

所以 Day 47 的核心不是安装，而是：

> **建立第一条真实可用的 serving 运行路径。**

---

## 2. vLLM setup 在验证什么

当你做 Day 47 时，实际上在验证以下几层：

### 1）硬件层
- GPU 是否可见
- CUDA 是否正常
- 显存是否足以加载目标模型

### 2）运行时层
- Python 环境是否干净
- vLLM 是否能正常启动
- 依赖版本是否冲突

### 3）模型层
- 所选模型是否支持当前后端
- tokenizer / 权重是否可正常加载
- precision 设置是否合适

### 4）服务层
- 服务进程是否能真正监听端口
- 是否可以响应一个测试请求

如果这四层都过了，setup 才算真正成功。

---

## 3. 为什么“选哪个模型先跑”很重要

Day 47 很容易犯的错误是：
- 一上来就选一个超大模型
- 环境还没验证好，先把系统压垮

更合理的策略通常是：

> **先用一个更小、已知兼容的模型把服务链路跑通，再逐步升级。**

因为 setup 阶段的目标不是追求最强效果，而是验证：
- 服务能启动
- 请求能通
- 日志能看
- 显存占用在预期内

这和 Day 37 做 RAG v1 的逻辑很像：

> **先跑通，再增强。**

---

## 4. setup 最常见的失败点

### 1）CUDA / 驱动不匹配
表现为：
- 启动时报错
- GPU 不可见
- 模型加载失败

### 2）显存不足
表现为：
- 服务启动时 OOM
- 推理阶段直接崩掉

### 3）模型不兼容
有些模型结构、权重格式、tokenizer 或配置与当前 vLLM 版本不完全匹配。

### 4）端口和服务路径问题
服务起来了，但：
- 端口没开放
- 请求没打对
- 客户端格式不兼容

这些问题都很常见，所以 setup 阶段一定要以“定位问题”为中心，而不是只盯着安装命令。

---

## 5. 为什么 setup 阶段也要记录观测信息

一个成熟的工程习惯是：

> setup 成功时，也要记录环境状态。

至少建议记录：
- 使用的模型名
- 模型大小
- GPU 型号
- 显存占用
- 是否启用 fp16 / bf16 / quantization
- 服务启动日志摘要
- 一次最小请求是否成功

因为这些信息在后面 benchmark（Day 49–50）和问题定位时都非常有用。

换句话说：

> **setup 不是一次性的杂活，而是后续实验的基准点。**

---

## 6. Day 47 和后面几天的关系

Day 47 是 Week 7 后面内容的入口。

### Day 48
你要测试 API 和 streaming

### Day 49–50
你要跑 benchmark

### Day 51+
你会继续进入 quantization / deployment trade-off

也就是说，如果 Day 47 没搭稳，后面所有结果都会不可靠。

所以 Day 47 的真正价值是：

> **给 serving 实验建立一块稳定地基。**

---

## 7. 今天最该记住的 5 句话

1. **vLLM setup 不是安装命令，而是在验证 serving 运行链路。**
2. **setup 必须同时通过硬件层、运行时层、模型层和服务层检查。**
3. **先用小模型跑通链路，比一开始追求大模型更合理。**
4. **setup 失败最常见的问题来自 CUDA、显存、兼容性和服务路径。**
5. **Day 47 是后续 API 测试和 benchmark 的基准地基。**

## 9. 一句话总结

> vLLM setup 的本质，不是把一个依赖装上去，而是建立并验证一条完整、可重复、可观测的大模型 serving 运行路径，为后续 API 测试、benchmark 和部署优化打下稳定基础。
