# Day 48 — vLLM API Testing

## 1. 为什么 setup 成功之后还要专门做 API testing

Day 47 解决的是：
- 服务能不能启动
- 模型能不能加载
- 最小请求能不能通

但这还远远不够。

因为在真实系统里，你不只是“本地跑一下推理”，而是需要：

> **让别的程序、脚本、前端、服务稳定调用这个模型。**

这就是 Day 48 的任务：
- 验证服务接口
- 验证请求格式
- 验证 streaming 行为
- 验证客户端兼容性

所以 Day 48 的重点不是模型，而是：

> **模型服务的接口契约是否可靠。**

---

## 2. API testing 在测什么

API testing 不是“发一个请求看看回了没”这么简单。

它至少应该验证：

### 1）请求格式
- prompt 字段怎么传
- model 字段是否正确
- 参数名称是否匹配

### 2）响应格式
- 返回 JSON 结构是什么
- 是否和客户端预期一致
- 错误时返回是否可解析

### 3）streaming 行为
- token 是否按流逐步返回
- 客户端是否能实时消费
- 流结束标志是否明确

### 4）兼容性
- 是否兼容 OpenAI 风格请求
- 是否能被现有 SDK / 前端轻松接入

这意味着 Day 48 在做的是：

> **把模型服务变成一个真正可接入的接口。**

---

## 3. 为什么 OpenAI-compatible API 很重要

现实工程里，很多应用并不想为每个推理后端都写一套全新客户端。

所以 OpenAI-compatible API 的价值非常大：
- 现有工具链更容易复用
- 前端 / 后端接入成本更低
- 很多 SDK 可以直接使用

这就是为什么 vLLM 的兼容接口很关键。

因为它的意义不只是“模仿一个格式”，而是：

> **降低模型服务接入门槛。**

---

## 4. streaming 为什么要单独测试

LLM 产品里，streaming 体验非常重要。

如果没有 streaming：
- 用户要等完整回答生成完才看到结果
- 主观延迟感明显更差

如果 streaming 可用：
- 用户更快看到首 token
- 产品体验更自然
- 对长回答尤其重要

所以 Day 48 必须明确区分：
- 服务能返回完整结果
- 服务能稳定流式返回结果

这两件事不是一回事。

---

## 5. API testing 的常见失败点

### 1）请求字段不兼容
比如客户端发的是一种格式，服务端需要另一种格式。

### 2）streaming 事件格式不稳定
- chunk 格式不一致
- 结尾标志不明确
- 前端消费困难

### 3）错误处理不规范
- 返回不是标准 JSON
- 错误信息难读
- 客户端无法做 fallback

### 4）超时与长请求问题
- 长 prompt 或长 generation 导致超时
- 客户端与服务端 timeout 配置不一致

所以 Day 48 本质上也是一次系统健壮性测试。

---

## 6. Day 48 为什么是产品化入口

如果 Day 47 是 serving 可运行，
那么 Day 48 就是 serving 可接入。

这是一个非常关键的区别。

### 可运行
你自己能调起来

### 可接入
别的系统也能稳定调用

只有到了“可接入”这一层，你才真正开始具备产品化基础。

---

## 7. 今天最该记住的 5 句话

1. **API testing 的目标是验证模型服务是否真正可接入。**
2. **请求格式、响应格式、错误处理和 streaming 都需要单独验证。**
3. **OpenAI-compatible API 的价值在于降低接入成本。**
4. **streaming 成功与完整响应成功不是同一件事。**
5. **Day 48 是从“模型能跑”走向“服务能接”的关键一步。**
