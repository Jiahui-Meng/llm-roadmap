# Day 83 — Repo Structure Cleanup

## 1. 为什么 Day 83 不是“整理文件”这么简单

到 Week 12，你的仓库里已经有：
- notes
- code
- experiments
- projects
- reports
- bridges
- assets

如果这些内容只是不断堆积，即使每个文件都不错，整体仓库也会慢慢失去可读性。

所以 Day 83 的重点不是整理桌面，而是：

> **把一个长期积累的学习仓库，收束成结构清晰的工程作品仓库。**

---

## 2. 为什么仓库结构本身就是作品的一部分

当别人打开你的 repo，第一印象往往不是某个算法细节，而是：
- 目录是不是清楚
- 命名是不是一致
- README 是否能引路
- 文件是否有明确归属

也就是说，仓库结构本身就在传达：
- 你有没有工程组织能力
- 你是否能长期维护项目
- 你的思路是否清晰

所以 repo structure cleanup 不是边角工作，而是：

> **作品可读性工程。**

---

## 3. cleanup 的目标到底是什么

Day 83 的目标不是“看起来更整齐”，而是：
- 降低理解成本
- 提升导航效率
- 减少重复与孤岛文件
- 统一命名规则
- 让项目边界更清楚

换句话说：

> **cleanup 是在降低未来读者和未来你自己的认知负担。**

---

## 4. 常见需要清理的问题

### 1）命名不一致
有的 day 用缩写，有的用全称，有的风格不同。

### 2）文件归属不清
某些脚本不知道应该在 `code/`、`experiments/` 还是 `projects/`。

### 3）中间文件污染
日志、缓存、临时文件混进仓库。

### 4）README 缺失
目录存在，但没人知道它是干什么的。

这些都会显著降低仓库可展示性。

---

## 5. 为什么 cleanup 对 portfolio 尤其重要

portfolio repo 和纯学习草稿 repo 不一样。

portfolio repo 需要：
- 外部可读
- 外部可导航
- 外部可展示

也就是说，它不是只服务你自己，而是面向：
- 面试官
- 合作者
- 未来雇主
- 公开读者

所以 Day 83 是在做一次角色切换：

> **从个人学习记录，转向公开工程作品。**

---

## 6. 今天最该记住的 5 句话

1. **repo cleanup 不是整理文件，而是在整理作品。**
2. **仓库结构本身就在展示工程组织能力。**
3. **cleanup 的目标是降低理解成本，而不只是美观。**
4. **命名统一、归属清晰、README 完整是关键。**
5. **Day 83 标志着仓库从学习记录走向 portfolio。**
