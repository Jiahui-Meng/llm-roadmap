# Day 74 — Retry and Reliability

## 1. 为什么 agent 在真实环境中必须考虑 reliability

只要你的 agent 依赖外部工具，它就迟早会遇到这些问题：
- 搜索接口超时
- 网页抓取失败
- 检索返回为空
- 模型生成不稳定
- 某一步输出格式不合法

如果系统没有 reliability 设计，这些小问题很容易放大成：
- 整条流程失败
- 用户什么也拿不到
- 体验非常脆弱

所以 Day 74 的重点是：

> **让 agent 不只“有时能跑”，而是“遇到波动时也能尽量完成任务”。**

---

## 2. retry 在解决什么问题

很多失败不是逻辑性失败，而是偶发性失败：
- 网络抖动
- 远程服务短暂错误
- 临时超时
- 对方接口瞬时不稳定

这类失败，如果立刻放弃，系统会显得很脆。

retry 的价值在于：
- 给系统一次恢复机会
- 把瞬时故障和真实故障区分开

所以 retry 不是“硬撑”，而是：

> **用有限重试提高系统韧性。**

---

## 3. fallback 在解决什么问题

有些失败不是重试几次就能好。

比如：
- 外部 search API 不可用
- scrape 网站结构完全变了
- summarize 模型持续失败

这时更重要的是：

> **系统能不能退化到一个次优但可接受的结果。**

例如：
- search 挂了 → 只用本地 retrieval
- scrape 挂了 → 跳过该链接，继续其他来源
- summarize 挂了 → 返回证据列表而不是完整摘要

fallback 的本质就是：

> **不要因为一个节点失败，就让整个用户价值归零。**

---

## 4. 为什么 reliability 不只是 retry + fallback

真正的 reliability 设计还包括：
- 错误分类
- timeout 设置
- 幂等性考虑
- 对失败路径的可观测性
- 用户可接受的降级输出

也就是说，Day 74 并不是“加个 tenacity 就完事”，而是：

> **系统性思考失败如何被感知、处理和降级。**

---

## 5. citation support 为什么也属于 reliability 的一部分

看起来 citation 更像质量问题，但在 research agent 里，它其实和 reliability 强相关。

因为即使系统不能给出很完整的总结，只要还能给出：
- 搜到的来源
- 证据片段
- 基本引用

用户依然能获得价值。

这说明：

> **可靠系统不一定每次都给完美答案，但应该尽量给有用且可验证的结果。**

这也是 citation support 被放在 Day 74 的原因之一。

---

## 6. 为什么 Day 74 是从“可跑”走向“可用”的关键日

前几天你做的是：
- 工具
- workflow
- trace

Day 74 则是在问：

> **如果现实世界不配合，这个系统还能不能维持基本可用？**

这非常关键，因为真实世界永远不稳定。

所以 Day 74 标志着 agent 系统进入一个更成熟阶段：
- 不只关注 happy path
- 也开始设计 unhappy path

---

## 7. 今天最该记住的 5 句话

1. **真实 agent 系统一定会遇到外部不稳定性。**
2. **retry 用来应对瞬时故障，fallback 用来应对持续故障。**
3. **reliability 不是加几个重试，而是系统性设计失败路径。**
4. **citation 也是 reliability 的一部分，因为它能在降级时保留用户价值。**
5. **Day 74 标志着系统从“可跑”走向“可用”。**

## 9. 一句话总结

> Retry and reliability 的本质，是为 agent 设计面对真实世界不稳定性的应对机制：瞬时故障可重试、持续故障可降级、结果尽量可验证，从而让系统在不完美环境中依然保持基本可用。
